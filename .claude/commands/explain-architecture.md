# Explain Architecture

You are helping a developer understand the architecture of their codebase by scanning source files, building a dependency graph from import statements, and generating a visual diagram.

**Arguments:** $ARGUMENTS

Parse the arguments:
- First positional argument (if any) = root path to scan (default: current directory)
- `--format <mermaid|plantuml|json>` = output format (default: mermaid)
- `--output <file>` = write result to this file instead of terminal
- `--depth <N>` = limit graph to N levels deep from root modules
- `--include-external` = include third-party packages as nodes
- `--preview` = always print to terminal even if --output is set

---

## Step 1 — Resolve target directory

Determine the root directory to scan:
- If a path argument was provided, resolve it relative to the project root
- Otherwise, use the current working directory
- Confirm the path exists

---

## Step 2 — Scan source files

Recursively find all source files in the target directory. Include:
- **TypeScript / JavaScript**: `.ts`, `.tsx`, `.js`, `.jsx`, `.mts`, `.mjs`
- **Python**: `.py`
- **Go**: `.go`

Exclude:
- `node_modules/`, `.git/`, `dist/`, `build/`, `out/`, `__pycache__/`, `.next/`, `coverage/`, `vendor/`
- Test files (`.test.ts`, `.spec.ts`, `_test.go`, `test_*.py`) unless they are the only files present
- Declaration files (`.d.ts`)

For each file, record:
- Relative path from scan root
- Language
- Top-level group = first meaningful subfolder (e.g. `src/controllers/auth.ts` → group `controllers`). Strip a leading `src/`, `lib/`, or `app/` prefix if present before grouping.

---

## Step 3 — Parse imports

For each source file, extract its import statements to identify dependencies on other files in the project.

**TypeScript / JavaScript** — match:
- `import ... from 'specifier'`
- `import('specifier')`
- `require('specifier')`
- `export ... from 'specifier'`

**Python** — match:
- `import module`
- `from module import ...`

**Go** — match:
- `import "package"` (single)
- `import (` block with one `"package"` per line `)`

For each specifier:
- If it starts with `.` or `..` → it is **internal**. Resolve the path relative to the importing file and find the matching scanned file.
- Otherwise → it is **external** (a third-party package). Keep only the root package name (e.g. `@org/pkg` or `express`).

Build a list of edges: `(source file path) → (target file path or external package name)`.

---

## Step 4 — Build the dependency graph

**File-level graph**: nodes are individual files, edges are resolved imports.

**Module-level graph** (default): collapse the file-level graph by group.
- A node = a group name (e.g. `controllers`, `services`, `utils`)
- An edge = at least one file in group A imports at least one file in group B
- Deduplicate edges (A → B appears only once)
- Remove self-loops (A → A)

If `--include-external` is not set, drop all external package nodes and their edges.

**Cycle detection**: run a DFS to find strongly connected components. If cycles exist, note them clearly.

**Depth filtering**: if `--depth N` is set, run a BFS from all root nodes (nodes with no incoming edges) and keep only nodes reachable within N hops.

---

## Step 5 — Generate the diagram

Based on the `--format` flag:

### Mermaid (default)

```
graph TD
    controllers["Controllers"]
    services["Services"]
    repositories["Repositories"]
    utils["Utils"]

    controllers --> services
    services --> repositories
    repositories --> utils
```

- Use `graph TD` (top-down) for most projects; use `graph LR` (left-right) if the graph is wide
- Quote node labels that contain special characters
- Highlight cyclic edges with a comment `%% CYCLE`

### PlantUML

```
@startuml
[Controllers] --> [Services]
[Services] --> [Repositories]
[Repositories] --> [Utils]
@enduml
```

### JSON

```json
{
  "nodes": ["controllers", "services", "repositories", "utils"],
  "edges": [
    ["controllers", "services"],
    ["services", "repositories"],
    ["repositories", "utils"]
  ],
  "cycles": []
}
```

---

## Step 6 — Format the output

Wrap the diagram in a Markdown document:

```markdown
# Project Architecture

> Generated by /explain-architecture · <date>

## Module Overview

| Module | Files | Description |
|--------|-------|-------------|
| controllers | 3 | ... |
| services | 5 | ... |

## Dependency Graph

\`\`\`mermaid
graph TD
    ...
\`\`\`

## Notes

- X modules · Y dependencies
- Cycles detected: (list or "none")
- External deps excluded (use --include-external to show them)
```

---

## Step 7 — Output

- If `--output <file>` is provided: write the Markdown document to that file and confirm the path.
- If `--preview` is set (or no `--output`): print the full output to the terminal.
- If both `--output` and `--preview` are set: do both.
- Always print a short terminal summary regardless:
  ```
  ✓ Scanned 42 files across 6 modules
  ✓ Found 18 dependencies, 0 cycles
  ✓ Written to architecture.md
  ```

---

## Edge cases

- **Circular dependencies**: flag them in the Notes section and mark the edges in the diagram.
- **Single-file projects**: show the file-level graph instead of collapsing to modules.
- **Large repos (> 200 files)**: warn the user and suggest using `--depth` or scanning a subdirectory.
- **Barrel / index files**: treat `index.ts` as transparent — use the folder name as the node, not the index file itself.
- **Multiple languages in one repo**: generate one graph per language or merge them (prefer merge when imports cross languages via generated bindings).
- **No imports found**: report this clearly and suggest checking if the path or file extensions are correct.
