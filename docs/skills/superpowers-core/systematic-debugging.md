# 系统性调试

> 随机修复浪费时间并引入新 Bug，快速补丁掩盖根本问题 -- 必须先找到根本原因，再尝试修复。

## 概述

`superpowers:systematic-debugging` 技能定义了一套严格的四阶段调试流程。它的核心原则是：**永远先找根本原因再修复，症状修复等于失败**。

该技能解决的问题是：开发者在遇到 Bug 时往往倾向于"试试改这个看看行不行"，这种随机修复方式不仅浪费时间，还容易引入新的 Bug。系统性调试方法强制要求理解问题后再动手。

### 铁律

```
没有完成根本原因调查，就不能提出修复方案
```

如果你没有完成阶段 1，你就不能提出任何修复。

### 实际效果对比

| 指标 | 系统性方法 | 随机修复方法 |
|------|-----------|-------------|
| 修复时间 | 15-30 分钟 | 2-3 小时反复折腾 |
| 一次修复成功率 | 95% | 40% |
| 引入新 Bug | 几乎为零 | 常见 |

## 使用场景

用于任何技术问题：
- 测试失败
- 生产环境 Bug
- 意外行为
- 性能问题
- 构建失败
- 集成问题

**尤其在以下情况时更要使用：**
- 时间紧迫（紧急情况更容易让人冲动猜测）
- "一个快速修复"看起来很明显
- 已经尝试了多次修复
- 上一次修复没有生效
- 你不完全理解问题

**以下情况也不要跳过：**
- 问题看起来很简单（简单 Bug 也有根本原因）
- 你很赶时间（系统性方法比反复折腾更快）
- 领导要求立即修好（系统性方法最终更快）

## 用法

调用方式：`/superpowers:systematic-debugging`

该技能在以下场景应被调用：
- 遇到任何 Bug、测试失败或意外行为时
- 在提出修复方案之前
- 前一次修复未生效时

## 四阶段调试流程

### 阶段 1：根本原因调查

**在尝试任何修复之前：**

#### 1. 仔细阅读错误信息

- 不要跳过错误或警告
- 错误信息往往包含确切的解决方案
- 完整阅读堆栈追踪
- 记录行号、文件路径、错误代码

#### 2. 稳定复现

- 能可靠地触发问题吗？
- 具体步骤是什么？
- 每次都会发生吗？
- 如果无法复现，收集更多数据，不要猜测

#### 3. 检查最近变更

- 什么改动可能导致了这个问题？
- 检查 git diff、最近的提交
- 新的依赖、配置变更
- 环境差异

#### 4. 在多组件系统中收集证据

当系统有多个组件时（如 CI -> 构建 -> 签名，API -> 服务 -> 数据库），在提出修复前需要添加诊断信息：

```bash
# 对每个组件边界：
# - 记录进入组件的数据
# - 记录离开组件的数据
# - 验证环境/配置传播
# - 检查每层的状态

# 示例：多层系统
# 层 1：工作流
echo "=== Secrets available in workflow: ==="
echo "IDENTITY: ${IDENTITY:+SET}${IDENTITY:-UNSET}"

# 层 2：构建脚本
echo "=== Env vars in build script: ==="
env | grep IDENTITY || echo "IDENTITY not in environment"

# 层 3：签名脚本
echo "=== Keychain state: ==="
security list-keychains

# 层 4：实际签名
codesign --sign "$IDENTITY" --verbose=4 "$APP"
```

这能揭示**哪一层失败了**（例如：secrets -> workflow 正常, workflow -> build 失败）。

#### 5. 追踪数据流

当错误深藏在调用栈中时：

- 坏值从哪里来？
- 谁用坏值调用了这个函数？
- 继续向上追踪直到找到源头
- 在源头修复，而非在症状处修复

### 阶段 2：模式分析

**在修复前找到规律：**

1. **找到可工作的示例** -- 在同一代码库中找到类似的、能正常工作的代码
2. **对比参考实现** -- 如果在实现某个模式，完整阅读参考实现（不要略读）
3. **识别差异** -- 正常和异常之间有什么不同？列出每个差异，无论多小
4. **理解依赖** -- 这个组件需要什么其他组件？什么设置、配置、环境？

### 阶段 3：假设与测试

**使用科学方法：**

1. **形成单一假设** -- 明确陈述："我认为 X 是根本原因，因为 Y"
2. **最小化测试** -- 做最小的可能变更来测试假设，每次只改一个变量
3. **先验证再继续** -- 生效了？进入阶段 4。没生效？形成新假设。不要叠加修复
4. **坦诚不知道** -- 说"我不理解 X"，不要假装知道

### 阶段 4：实施修复

**修复根本原因，而非症状：**

1. **创建失败测试用例** -- 最简单的复现方式，自动化测试优先。使用 `superpowers:test-driven-development` 技能
2. **实施单一修复** -- 针对已识别的根本原因，每次一个变更，不要捆绑"顺手"的改进
3. **验证修复** -- 测试通过了吗？其他测试没破坏吧？问题真正解决了吗？
4. **修复无效时**：
   - 已尝试 < 3 次：返回阶段 1，用新信息重新分析
   - **已尝试 >= 3 次：停下来质疑架构**（见下文）

#### 3 次修复失败后：质疑架构

**指示架构存在问题的模式：**
- 每次修复都揭示新的共享状态/耦合/不同位置的问题
- 修复需要"大规模重构"才能实施
- 每次修复在其他地方产生新症状

**停下来质疑基本假设：**
- 这个模式从根本上是否合理？
- 我们是否因为惯性而坚持使用它？
- 应该重构架构还是继续修复症状？

**在尝试更多修复之前与用户讨论。** 这不是假设失败，而是架构错误。

## 示例

### 示例 1: 基本调试流程

```
用户: 用户注册时报 500 错误

Claude: 使用 systematic-debugging 技能进行调试。

阶段 1 -- 根本原因调查：
1. 阅读错误日志：
   TypeError: Cannot read property 'toLowerCase' of undefined
   at validateEmail (src/validators.js:15)

2. 检查 validateEmail 第 15 行：
   email.toLowerCase() -- email 参数为 undefined

3. 追踪调用链：
   registerUser -> validateInput -> validateEmail
   发现 validateInput 传入的是 req.body.Email（大写 E）
   而 validateEmail 期望 req.body.email（小写 e）

4. 根本原因：前端表单字段名更改后后端未同步更新

阶段 4 -- 修复：
[编写失败测试 -> 修复字段名映射 -> 验证通过]
```

### 示例 2: 多组件系统调试

```
Claude: 使用 systematic-debugging 技能进行调试。

阶段 1 -- 收集证据：
在每个组件边界添加诊断日志...

层 1 (API Gateway): 请求正确接收，headers 完整
层 2 (Auth Service): JWT token 验证通过
层 3 (User Service): 收到请求但 userId 为 null
层 4 (Database): 查询因 null userId 失败

根本原因：Auth Service 返回的 token payload 使用
"sub" 字段，但 User Service 读取的是 "userId" 字段。
```

### 示例 3: 3 次修复失败后质疑架构

```
Claude: 已尝试 3 次修复：
- Fix 1: 修正缓存 key 格式 -> 揭示了竞态条件
- Fix 2: 添加锁机制 -> 揭示了死锁问题
- Fix 3: 改用乐观锁 -> 揭示了数据不一致

每次修复都在不同位置暴露新问题。这表明缓存架构
本身可能有根本性问题。

建议：
A) 重新设计缓存层架构（推荐）
B) 继续逐个修复症状（不推荐）
C) 暂时禁用缓存，评估性能影响

需要您的决定才能继续。
```

## 红旗信号 -- 立即停止

如果你发现自己在想：
- "先快速修一下，之后再调查"
- "试试改 X 看看行不行"
- "同时改多个地方，跑测试"
- "跳过测试，手动验证"
- "可能是 X，让我修一下"
- "我不完全理解但这也许行"
- "再试一次修复"（已经试了 2 次以上）

**所有这些都意味着：停下来。返回阶段 1。**

## 常见合理化借口

| 借口 | 现实 |
|------|------|
| "问题很简单，不需要流程" | 简单问题也有根本原因，流程对简单 Bug 很快 |
| "紧急情况，没时间走流程" | 系统性调试比猜测式折腾更快 |
| "先试这个，再调查" | 第一次修复设定了模式，一开始就做对 |
| "修复确认后再写测试" | 未经测试的修复不可靠，先写测试才能证明 |
| "同时修多个节省时间" | 无法分离哪个生效了，还会引入新 Bug |
| "参考文档太长，我适配一下" | 部分理解保证出 Bug，完整阅读 |

## 辅助技术

技能目录中包含的支撑技术：

- **root-cause-tracing.md** -- 通过调用栈反向追踪 Bug 到原始触发点
- **defense-in-depth.md** -- 找到根本原因后在多层添加验证
- **condition-based-waiting.md** -- 用条件轮询替代随意的超时等待

## 注意事项

- **这是刚性技能** -- 必须严格遵循四阶段流程，不得跳过或简化
- **阶段 1 是硬性门控** -- 没有完成根本原因调查，不能提出修复方案
- **3 次修复规则** -- 3 次以上修复失败表明架构问题，必须停下来讨论
- **证据优先** -- 每个结论都必须有证据支撑，不接受"可能是"或"应该是"
- **与其他技能的关系**：
  - 调用时机：遇到任何 Bug、测试失败、意外行为时
  - 阶段 4 调用：`superpowers:test-driven-development` -- 创建失败测试用例
  - 完成后调用：`superpowers:verification-before-completion` -- 验证修复是否真正生效
  - 通常先于实现类技能被调用（流程类技能优先级更高）
