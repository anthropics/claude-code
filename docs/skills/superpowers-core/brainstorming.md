# 头脑风暴 -- 将创意转化为设计方案

> 在任何创造性工作之前，通过协作对话将模糊的想法转化为完整的设计规格，确保在编码前充分理解需求。

## 概述

`superpowers:brainstorming` 是 Superpowers 插件中最重要的前置技能之一。它的核心理念是：**在写任何代码之前，必须先完成设计并获得用户批准**。无论项目看起来多么简单，都必须经过设计流程 -- 因为"简单"项目正是未经检验的假设最容易导致浪费的地方。

该技能通过自然的协作对话，帮助用户：
1. 梳理项目上下文
2. 逐一澄清需求
3. 探索 2-3 种实现方案
4. 形成完整的设计文档
5. 获得用户批准后过渡到实施计划

### 硬性门控

在用户批准设计之前，**严禁**调用任何实现类技能、编写任何代码、搭建任何项目框架。这适用于所有项目，无论其看起来多么简单。

## 使用场景

- 创建新功能或新组件时
- 构建新应用或新项目时
- 修改现有功能的行为时
- 进行任何涉及创意决策的工作时
- 即将进入 Plan Mode 但尚未进行头脑风暴时

## 用法

调用方式：`/superpowers:brainstorming`

该技能在以下情况下应被自动触发：
- 用户请求创建、构建、添加功能时
- 用户描述一个需要设计决策的任务时
- 即将进入计划模式（Plan Mode）但尚未进行头脑风暴时

## 完整流程

该技能定义了一个严格的检查清单，必须按顺序完成：

1. **探索项目上下文** -- 检查文件、文档、最近的提交记录
2. **提供可视化伴侣**（如果涉及视觉问题）-- 这是单独的一条消息，不与其他问题混合
3. **逐一提问澄清** -- 每条消息只问一个问题，理解目的、约束和成功标准
4. **提出 2-3 种方案** -- 包含权衡分析和推荐建议
5. **展示设计方案** -- 按复杂度分节展示，每节获得用户确认
6. **编写设计文档** -- 保存到 `docs/superpowers/specs/YYYY-MM-DD-<topic>-design.md` 并提交
7. **规格自审** -- 检查占位符、矛盾、歧义、范围
8. **用户审核规格** -- 请用户审核规格文件后再继续
9. **过渡到实施** -- 调用 `superpowers:writing-plans` 技能创建实施计划

## 示例

### 示例 1: 基本用法 -- 新功能设计

```
用户: 帮我给应用添加一个暗色模式

Claude: 使用 brainstorming 技能来设计暗色模式功能。

让我先看看当前项目的结构和样式方案...
[探索项目上下文]

第一个问题：暗色模式需要支持哪些切换方式？
A) 手动切换（用户点击按钮）
B) 跟随系统设置自动切换
C) 两者都支持（手动切换 + 自动跟随）

用户: C

[继续逐一提问，直到理解所有需求...]
[提出 2-3 种实现方案...]
[分节展示设计并获得批准...]
[编写设计文档并提交...]
[调用 writing-plans 创建实施计划]
```

### 示例 2: 高级用法 -- 大型项目分解

当用户描述的项目涉及多个独立子系统时（例如"构建一个包含聊天、文件存储、计费和分析的平台"），技能会：

1. **立即标记范围过大** -- 不会花时间细化一个需要分解的项目的细节
2. **帮助用户分解子项目** -- 确定独立的模块、它们之间的关系、构建顺序
3. **对第一个子项目进行完整的头脑风暴** -- 每个子项目走独立的 规格 -> 计划 -> 实施 周期

### 示例 3: 在现有代码库中工作

```
用户: 我需要重构用户认证模块

Claude: 使用 brainstorming 技能来设计认证模块重构。

让我先探索当前的认证模块结构...
[查看现有文件、模式、依赖关系]

在现有代码库中，我会：
- 先了解当前结构，再提出更改方案
- 遵循已有的代码模式和约定
- 只在影响当前工作的地方提出改进（如过大的文件、不清晰的边界）
- 不提出无关的重构

第一个问题：当前认证模块最大的痛点是什么？
A) 代码耦合度高，难以测试
B) 不支持多种认证方式
C) 性能问题
D) 其他（请描述）
```

## 设计原则

### 隔离与清晰

- 将系统拆分为职责单一、接口清晰、可独立理解和测试的小单元
- 对每个单元，应能回答：它做什么、如何使用它、它依赖什么
- 如果不看内部实现就无法理解一个单元的功能，说明边界需要优化
- 更小、边界清晰的单元也更适合 AI 工作 -- AI 在能完整理解上下文时推理更准确

### 提问策略

- **每条消息只问一个问题** -- 不要用多个问题轰炸用户
- **优先使用选择题** -- 比开放式问题更容易回答
- **YAGNI 原则** -- 无情地移除不必要的功能
- **增量验证** -- 展示设计，获得批准后再继续

### 可视化伴侣

当预期后续问题涉及视觉内容（界面原型、布局对比、架构图）时，技能会提供一个基于浏览器的可视化伴侣。注意：

- 提供可视化伴侣必须是**独立的一条消息**，不与其他问题混合
- 即使用户接受了可视化伴侣，也要逐个问题判断是否需要视觉呈现
- 概念性问题用终端文本，视觉性问题用浏览器

## 注意事项

- **"这太简单了不需要设计"是一个反模式** -- 每个项目都需要经过设计流程，设计可以很简短，但必须存在且获得批准
- **设计文档必须保存并提交** -- 默认保存到 `docs/superpowers/specs/` 目录
- **规格自审是必需的** -- 检查占位符（TBD/TODO）、内部矛盾、范围是否合适、歧义性
- **用户审核是硬性门控** -- 必须等待用户批准规格后才能继续
- **与其他技能的关系**：
  - 前置：本技能是实施流程的起点
  - 后续：完成后只能调用 `superpowers:writing-plans`
  - 不得直接跳到实现类技能（如 frontend-design）
